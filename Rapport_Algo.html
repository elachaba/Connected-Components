I. Algorithme Depth-First-Search:¶

L'algorithme Depth-First Search (DFS) est une technique de recherche dans les graphes qui explore les branches aussi profondment que possible avant de revenir en arrire pour explorer d'autres branches.
Notre ide tait donc de reprsenter les points sous forme d'un graphe. Nous avons implment ce graphe en utilisant un dictionnaire en python, dont les cls reprsentaient les points, et donc les noeuds du graphe, et pour chaque cl, sa valeur tait un ensemble de points qui lui sont voisins, sous le critre de connectivit. Notre ide d'utiliser un dictionnaire tait pour minimiser le temps d'accs sachant que le cot de recherche d'une cl dans un dictionnaire est constant. Pour la mme raison nous avons opt pour la structure d'ensemble pour stocker les valeurs plutt qu'une liste.

II.Structure Union-Find¶

L'algorithme Union-Find est une structure de donnes qui permet de grer des ensembles disjoints et est particulirement utile pour les problmes de connectivit dans les graphes.
L'ide donc est d'avoir un point reprsentant pour chaque composante connexe et de stocker la taille de la composante dont ce point est le reprsentant.
On a initialis une liste de classes class_liste, o chaque point est considr comme une classe distincte (un singleton) au dbut. Il y a donc n classes, o n est le nombre de points, ensuite on parcourt chaque paire de points distincts dans la liste des points. Pour chaque paire de points, on vrifie s'ils sont voisins. On utilise donc une fonction find pour remonter au reprsentants des composantes connexes auquelles appartiennent les deux points considrs. Aprs, il ne reste que fusionner ces deux classes.

def find(parents, point):
    """Trouver la classe d'un point"""
    if parents[point] != point:
        parents[point] = find(parents, parents[point])
    return parents[point]

def union(parents, sizes, point1, point2):
    """Union two components in the union-find data structure"""
    parent_1 = find(parents, point1)
    parent_2 = find(parents, point2)
    if parent_1 != parent_2:
        if sizes[parent_1] < sizes[parent_2]:
            parent_1, parent_2 = parent_2, parent_1
        parents[parent_2] = parent_1
        sizes[parent_1] += sizes[parent_2]

def print_components_sizes(distance, points):
    """
    affichage des tailles triees de chaque composante
    """
    num_points = len(points)
    parents = list(range(num_points))
    sizes = [1] * num_points
    for i in range(1, num_points):
        for j in range(i):
            if points[i].distance_to(points[j]) <= distance:
                union(parents, sizes, i, j)
    class_sizes = {}
    for i in range(num_points):
        parent = find(parents, i)
        if parent not in class_sizes:
            class_sizes[parent] = 0
        class_sizes[parent] += 1
    class_list = list(class_sizes.values())
    class_list.sort(reverse=True)
    print(f"{class_list}")

Dtaillons la complexit de cet algorithme:
find: La complexit de la fonction find avec compression de chemin est $O(\alpha(n))$, o $\alpha(n)$ est la fonction inverse d'Ackermann. Cette fonction crot trs lentement.
union: La complexit de la fonction union dpend principalement de la fonction find, car elle l'appelle deux fois. La complexit de union est donc galement $O(\alpha(n))$.
print_components_sizes: Cette fonction a plusieurs parties :
1. Initialisation des parents et des tailles : $O(1)$.
2. Boucle double pour vrifier la distance entre les paires de points et appeler la fonction union : Il y a $O(n^2)$ paires de points, et chaque appel union a une complexit $O(\alpha(n))$, donc cette partie a une complexit totale de $O(n^2\,\alpha(n))$.
3. Calcul des tailles de classes et tri : $O(n\,\alpha(n))$ pour parcourir les parents et calculer les tailles, et $O(m\,log(m))$ pour trier les tailles des classes, avec m le nombre de composantes connexes.

La complexit globale de l'algorithme est donc domine par la boucle double et le tri, ce qui donne une complexit de $O(n^2\,\alpha(n) + m\,log(m))$. Dans le cas moyen et le pire cas, la complexit est gnralement domine par le terme $O(n^2\,\alpha(n))$, car le nombre de points peut tre suffisamment grand pour que le terme $m\,log(m)$ devienne ngligeable par rapport $n^2\,\alpha(n).$

Dans le meilleur des cas, si tous les points sont isols, alors la boucle double ne ferait aucune union. Toutefois, elle vrifierait toujours toutes les paires de points, ce qui reprsente $O(n^2)$ paires de points.
Ainsi, la complexit pour le meilleur cas serait domine par la boucle double, et le tri :$O(n^2)$ pour la boucle double et $O(n\,log(n))$ pour le tri. La complexit globale du meilleur cas serait donc $O(n^2 + n\,log(n))$, qui est domine par $O(n^2)$.

III. Structure Grid¶

L'ide principale de cet algorithme est de minimiser le nombre de points qu'on verifie chaque tape d'union. On rpartie le plan donc sous forme d'une grille de cellules de mme taille (dans ce cas ct = distance).
La fonction coord_cell calcule les coordonnes de la cellule laquelle appartient un point dans la grille. Chaque point est plac dans une cellule en fonction de ses coordonnes.
La fonction build_grid construit la grille en ajoutant les points dans leurs cellules correspondantes. Cette structure de donnes facilite la recherche des points voisins d'un point donn.
La fonction implement_union_find construit la structure de donnes Union-Find pour trouver les composantes connectes parmi les points en fonction de la distance spcifie. Elle commence par construire la grille, puis parcourt chaque cellule de la grille et ses cellules adjacentes pour vrifier si les points sont proches les uns des autres. Si deux points sont proches, ils sont unis dans la structure Union-Find.

def coord_cell(point, distance):
    """
    Given a point and distance,
    calculates the coordinates of the cell that the point belongs to in the grid.
    """
    abscisse = int(point.coordinates[0] // distance)
    ordonnee = int(point.coordinates[1] // distance)
    return abscisse, ordonnee


def build_grid(distance, points):
    """
    Constructs the grid data structure
    by putting points into their corresponding cells.
    """
    grid = {}
    for idx, point in enumerate(points):
        cell = coord_cell(point, distance)
        if cell not in grid:
            grid[cell] = set()
        grid[cell].add(idx)
    return grid

def implement_union_find(distance, points):
    """
    Given the list of points and the distance threshold,
    constructs and returns the Union-Find data structure
    to efficiently find connected components among the points
    based on the specified distance.
    """
    parents = list(range(len(points)))
    ranks = [0] * len(points)
    grid = build_grid(distance, points)

    for cell, point_indexes in grid.items():
        adj_cells = [
            (cell[0] + dx, cell[1] + dy) for dx in range(-1, 2)
            for dy in range(-1, 2)
        ]

        for idx in point_indexes:
            for adj_cell in adj_cells:
                if adj_cell in grid:
                    for j in grid[adj_cell]:
                        if idx != j and points[idx].distance_to(points[j]) <= distance:
                            union(parents, ranks, idx, j)

    return parents

Le but donc est de ne parcourir que les points dans les cellules adjacentes un point au lieu de parcourir tous le points du graphe chaque fois.
La complexit de cet algorithme dpend des entres, le nombre de points n, et la distance seuil d, qui contrlent la densit du graphe.
La fonction coord_cell a une complexit temporelle constante, car elle effectue un calcul simple pour dterminer les coordonnes de la cellule laquelle appartient un point.
La fonction build_grid a une complexit temporelle en $O(n)$, puisqu'elle parcourt tous les points une fois et effectue un travail constant pour chaque point en les ajoutant la grille.
Pour la fonction implement_union_find la complexit dpend du nombre de points n, de la distance seuil d. Dans le pire des cas, tous les points sont proches les uns des autres, et la complexit temporelle est domine par la recherche des voisins pour chaque point qui sera de l'ordre de $O(n^2\,\alpha(n))$. En moyenne, la complexit est beaucoup plus faible en raison de la structure de grille qui rduit considrablement le nombre de comparaisons de distances effectuer.
Le meilleur cas se produit encore lorsque les points sont suffisamment loigns les uns des autres, de sorte qu'aucun point n'est une distance infrieure ou gale la distance seuil d'un autre point. Dans ce cas, il n'y aurait aucune union effectuer, car il n'y aurait pas de composantes connectes. La complexit temporelle de la fonction implement_union_find serait alors domine par la construction de la grille. tant donn que chaque point est insr dans la grille une seule fois, la complexit temporelle serait $O(n)$ pour cette tape. La complexit temporelle totale de l'algorithme serait $O(n)$.

Comparaison entre les algorithmes¶

Dans cette section, nous allons comparer les trois algorithmes que nous avons prsents. Nous allons tracer les courbes de complexit en fonction du nombre de points et de la distance seuil. Nous nous baserons ensuite sur ces courbes pour comparer ces diffrents algorithmes.

I. Effet du nombre de points¶

Nous avons crit un programme qui gnre plusieurs fichiers avec des nombres de points differents et une distance fixe 0.1. Le but est d'analyser l'effet du nombre de points sur la performance de nos algorithmes. On a ensuite trac les courbes de complexit temporelle en fonction du nombre de points de l'entre.

Le comportement de l'algorithme DFS lorsque le nombre de point augmente est effectivement de $O(n^2)$. C'est un algorithme trs lent par rapport au deux autres. Ceci revient au fait que l'algorithme du DFS traite point par point, pour tous les points du graphe, alors que l'Union-Find permet de travailler sur des classes d'quivalences de points, et la grille restreint le domaine de recherche des voisins. En plus, la grille (grid), est effectivement une version amliore de l'Union-Find. Il est noter qu'on a limit le nombre de points 3000, pour d = 0.1, puisque l'algorithme DFS prend un temps assez grand pour s'xecuter.

II. Effet de la distance seuil¶

Nous avons fix le nombre de points pour le premier test 1200 points cause du temp d'exuction de l'algorithme du DFS. Pour comparer la performance des deux utilisations de la structure Union-Find, nous avons augmenter le nombre de points 3000 et tester sur plus de valeurs de distance.

Le DFS reste toujours le moins performant. Nanmoins, on remarque que la complexit de l'union-find est approximativement constante en fonction de la distance, ce qui lui permet d'tre plus performant que l'algorithme utilisant la structure grid au-del de $\approx 0.25$. La complexit de ce dernier tant dpendante de la distance, on explique cet effet remarqu par le fait que la taille des cellules augmente, et donc le nombre de points au sein d'une mme celulle augmente galement. Ainsi, on augmente le nombre de points dans une cellule et ainsi le nombre de comparaisons faire.
Par suite, on pourrait utiliser l'algorithme grid pour un nombre de points lev mais pour des distances infrieurs $0.25$, et utiliser l'algorithme de Union-Find pour des distances suprieures $0.25$.

Pistes d'amélioration¶

Nous proposons un algoithme utilisant la structure grid, mais cette fois-ci sans utiliser l'algorithme de Union-Find. On utilise plutt l'algorithme DFS pour la recherche des voisins dans les cellules adjacentes. On considre ainsi que les points au sein d'une cellule sont tous voisins, on les considre tous dans une seule composante connexe. Ainsi, chaque cellule constitue une partie connexe, et donc on ne traite plus des points mais des cellules. Ce qui nous permettrait de rduire le nombre de points visiter.

L'ide de l'algorithme est la suivante:

  1. On initialise un ensemble visited_cells pour suivre les cellules dj visites et une liste components pour stocker les tailles des composantes connectes.

  2. On parcourt toutes les cellules de la grille. Si une cellule n'a pas encore t visite, elle commence explorer les composantes connectes partir de cette cellule en utilisant une approche de recherche en profondeur (DFS) avec une pile stack.

  3. Pendant l'exploration, elle met jour la taille de la composante connecte en ajoutant le nombre de points de la cellule actuelle. Ensuite, elle examine les cellules adjacentes la cellule actuelle pour vrifier si elles sont connectes la cellule actuelle. Si une paire de points est trouve avec une distance infrieure ou gale la distance spcifie, la cellule adjacente est ajoute la pile pour tre explore plus tard.

  4. Une fois l'exploration termine, la taille de la composante connecte est ajoute la liste components.

  5. Et on itre ...

def print_components_sizes(distance, points):
    grid = build_grid(distance, points)
    visited_cells = set()
    components = []
    for cell in grid:
        if cell not in visited_cells:
            stack = [cell]
            length_composante = 0
            while stack:
                curr_cell = stack.pop()
                if curr_cell not in visited_cells:   
                    visited_cells.add(curr_cell)
                    adj_cells = [
                        (cell[0] + dx ,cell[1] + dy) for dx in [-1,1] for dy in [-1,1]]\
                        + [(cell[0]-1,cell[1]),(cell[0]+1,cell[1]),(cell[0],cell[1]-1),(cell[0],cell[1]+1)]\
                        + [(cell[0] + dx ,cell[1] + dy) for dx in [-2,2] for dy in [-1,0,1]]\
                        + [\
                            (cell[0]-1,cell[1]+2),\
                           (cell[0],cell[1]+2),\
                           (cell[0]+1,cell[1]+2),\
                           (cell[0]-1,cell[1]-2),\
                           (cell[0],cell[1]-2),\
                           (cell[0]+1,cell[1]-2)
                           ]
                    length_composante += len(grid[curr_cell])
                    for adj_cell in adj_cells:
                        if adj_cell in grid and adj_cell not in visited_cells:
                            marker = len(stack)
                            for idx_point in grid[curr_cell]:
                                for idx_adj_point in grid[adj_cell]:
                                    if points[idx_point].distance_to(points[idx_adj_point]) <= distance:
                                        if adj_cell not in stack:
                                            stack.append(adj_cell)
                                            break
                                if len(stack) > marker:
                                    break
            components.append(length_composante)
    components.sort(reverse=True)
    print(components)

Le point d'amlioration apport par cet algorithme est de diminuer encore le nombre de points parcourir. En effet, on commence par des composantes contenant plusieurs points connects et non pas un seul point, et pour chaque cellule, une fois on trouve un point d'une cellule adjacente qui est voisin un point de cette cellule, on fait un break et on ajoute le nombre de points de la cellule adjacente aux nombre de points total de la composante courante.
$\mathcal{NB}$: Tous les points d'une cellule sont voisins car les cellules sont choisies de sorte que leur diagonale soit gale la distance distance.
